## Summary

Phase 2.1 is a pipeline rewrite, not a UI change. Every script in `scripts/` feeds into a final output shape that the existing scoring engine and all Phase 2 UI pages already consume. The challenge is entirely in replacing the simulated trade generator (`fetch-trades.ts`) with real Alva API calls, and ensuring the mapping layer faithfully produces the same `RawTrade` shape.

The critical architectural discovery: **the Alva SDK is a sandboxed V8 runtime, not an npm package.** You cannot `import` or `require('@alva/...')` directly in a Node.js `tsx` script. Instead, the pattern is to call Alva's REST API (`POST /api/v1/run` with `X-Alva-Api-Key`) to execute a script in the Alva cloud, which can then use Alva's `require('@arrays/...')` modules. Alternatively, Alva exposes raw data via discoverable SDK partition endpoints. The build script must either (a) POST a script to Alva's run endpoint and receive JSON back, or (b) call a higher-level Alva REST endpoint that surfaces senator trade data directly.

The mapping layer must bridge the Alva response fields (name, ticker, transactionDate, filingDate, transactionType, amount range) into the existing `RawTrade` interface fields. Company name and sector enrichment comes from either Alva's `equity_fundamentals` partition or the existing static `TICKERS` map already in `fetch-trades.ts`. Stock price data for return calculations uses Alva's `spot_market_price_and_volume` partition via the same REST API call pattern.

**Primary recommendation:** Call Alva's REST API from the build script using standard `fetch()`, executing Alva cloud scripts that use the `equity_ownership_and_flow` and `spot_market_price_and_volume` SDK modules. The mapping layer in `fetch-trades.ts` handles the transformation — no downstream changes needed.

### Pattern 1: Alva REST API Call from Node.js Build Script

**What:** POST to Alva's `/api/v1/run` with an inline script that uses the sandboxed SDK modules. The script runs in Alva's cloud and returns JSON output.
**When to use:** Fetching senator trades and stock prices from Alva SDK partitions.

```typescript
// Source: https://github.com/alva-ai/skills/blob/main/skills/alva/references/api-reference.md
const ALVA_ENDPOINT = process.env.ALVA_ENDPOINT ?? 'https://api-llm.prd.alva.ai'
const ALVA_API_KEY = process.env.ALVA_API_KEY

if (!ALVA_API_KEY) {
  console.error('[fetch-trades] ALVA_API_KEY not set — build cannot proceed')
  process.exit(1)
}

async function callAlvaScript(script: string): Promise<unknown> {
  const res = await fetch(`${ALVA_ENDPOINT}/api/v1/run`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Alva-Api-Key': ALVA_API_KEY!,
    },
    body: JSON.stringify({ code: script }),
  })
  if (!res.ok) {
    const text = await res.text()
    throw new Error(`Alva API error ${res.status}: ${text.slice(0, 300)}`)
  }
  return res.json()
}
```

### Pattern 5: Dynamic Season Window Calculation

**What:** Instead of hardcoded `SEASON_START`/`SEASON_END` constants, compute the 6-week window from the most recent trade date in the dataset.
**When to use:** In `generate-demo.ts` — update the season window constants.

```typescript
// Determine "most recent 3 months" from actual trade data
function computeSeasonWindow(trades: RawTrade[]): { start: Date; end: Date; weeks: { start: Date; end: Date }[] } {
  const dates = trades.map(t => new Date(t.tradeDate)).sort((a, b) => b.getTime() - a.getTime())
  const end = dates[0] // Most recent trade date
  const start = subMonths(end, 3) // 3 months back
  // Divide into 6 equal weekly windows
  const totalMs = end.getTime() - start.getTime()
  const weekMs = totalMs / 6
  const weeks = Array.from({ length: 6 }, (_, i) => ({
    start: new Date(start.getTime() + i * weekMs),
    end: new Date(start.getTime() + (i + 1) * weekMs),
  }))
  return { start, end, weeks }
}
```

### Anti-Patterns to Avoid

- **Importing Alva SDK modules directly in tsx scripts:** The `require('@arrays/...')` pattern only works inside Alva's V8 sandbox. A build script that tries to do this will crash with "Cannot find module."
- **Silent fallback to simulated data:** D-04 is explicit — if Alva is unreachable, `process.exit(1)` with a clear error. Remove the `tryQuiverAPI()` fallback pattern entirely.
- **Hardcoded season dates:** The current `SEASON_START = new Date('2025-10-01')` and `SEASON_END = new Date('2025-12-23')` must be replaced with computed values from real trade dates (D-03).
- **Fetching stock prices one-by-one synchronously:** Congressional trade datasets can have 100+ unique tickers. Batch price lookups in parallel with rate limiting — do not await each one serially.
- **Assuming Alva returns `bioguideId`:** Alva returns politician names, not Congress.gov bioguide IDs. The mapping layer must join Alva trade data to `_raw-politicians.json` (from `fetch-politicians.ts`) by normalizing politician names.

## Common Pitfalls

### Pitfall 1: Alva SDK Is Not an npm Package
**What goes wrong:** Developer installs `npx skills add https://github.com/alva-ai/skills` and expects to `import { getSenatorTrades } from '@alva/...'` in a tsx script. This fails — the npm install sets up Claude Code MCP tool context, not importable Node.js modules.
**Why it happens:** The Alva skill README implies SDK-style usage, but the runtime is a sandboxed V8 environment accessed via REST API.
**How to avoid:** Call Alva via `fetch()` to `POST /api/v1/run`. Pass the SDK calls as an inline code string. Parse the JSON response.
**Warning signs:** `Error: Cannot find module '@arrays/...'` in tsx output.

### Pitfall 2: Politician Name Matching Across Data Sources
**What goes wrong:** Alva returns `"Nancy Pelosi"` or `"Pelosi, Nancy"`. `fetch-politicians.ts` uses Congress.gov's `bioguideId`. The mapping must join these two datasets by name — which is fragile if name formats differ.
**Why it happens:** Alva doesn't expose Congress.gov bioguide IDs; those come from the Congress.gov API.
**How to avoid:** Build a normalized name lookup: lowercase, strip honorifics, split on comma and spaces. Build a `Map<string, bioguideId>` from `_raw-politicians.json` before processing Alva trades. Log unmatched names as warnings (don't fail — some politicians may not be in Congress.gov results).
**Warning signs:** Low politician count in final `politicians.json`, or "unknown_politician" exclusions dominating the build report.

### Pitfall 3: Amount Range String Format Varies
**What goes wrong:** Alva (or the underlying source) returns `"$1,001 - $15,000"` but the existing `AmountRange` type expects `"$1k-$15k"`. `score-all.ts` will throw on invalid amountRange values.
**Why it happens:** STOCK Act disclosure amounts are reported as dollar ranges; different APIs format them differently.
**How to avoid:** Write a robust `mapAmountRange()` function that parses the upper bound of the amount string and maps to the existing `AmountRange` union. Use numeric thresholds: `<= 15000 → '$1k-$15k'`, `<= 50000 → '$15k-$50k'`, etc. Test with known edge cases.
**Warning signs:** `score-all.ts` crashing with "invalid amountRange" or trades with 0 points from unrecognized ranges.

### Pitfall 4: Missing Ticker in Static Map Causes Sector = undefined
**What goes wrong:** Real congressional trades include tickers not in the existing `TICKERS` map. `sector` field becomes `undefined`, which breaks filtering in the UI (Phase 2 pages).
**Why it happens:** The static map only covers ~60 common tickers. Real congressional portfolios include hundreds of tickers.
**How to avoid:** (a) Use Alva `equity_fundamentals` for sector lookup on unknown tickers, OR (b) Use a larger static ticker-to-sector mapping (GICS sector codes + manual mapping), OR (c) Fall back to `'Other'` for unknown tickers with a console warning.
**Warning signs:** UI filter dropdowns missing sectors, or `undefined` values in `trades.json`.

### Pitfall 5: Hardcoded Season Dates Break with Real Data
**What goes wrong:** `generate-demo.ts` currently has `NUM_WEEKS = 6` and `weeklyPoints[week-1]` indexing. If real trade data doesn't span exactly 6 equal weeks, weekly point buckets will be empty or misaligned.
**Why it happens:** Simulated trades were generated exactly within the hardcoded season. Real trades span an arbitrary date range.
**How to avoid:** Compute the 6-week window dynamically from `scores[].tradeDate` values (D-03). Update `score-all.ts` to bucket trades into the dynamic week windows when computing `weeklyPoints`.
**Warning signs:** All `weeklyPoints` arrays containing all zeros, or matchup scores all being 0.

### Pitfall 6: Alva API Rate Limits on Bulk Price Lookups
**What goes wrong:** Fetching prices for 200+ unique tickers in rapid succession triggers rate limiting, causing partial data and inconsistent scoring.
**Why it happens:** Each `callAlvaScript()` invocation is an API call; bulk data calls may also have limits.
**How to avoid:** Batch price lookups: fetch multiple tickers per Alva script invocation where possible. Implement a simple retry with 1-2 second delay on 429 responses. Cache results by ticker to avoid duplicate lookups.
**Warning signs:** HTTP 429 responses in build output, large number of trades with missing price data.

### Amount Range Normalizer (Mapping Layer)
```typescript
// Source: Industry standard - verified against Finnhub/AInvest response formats
// Uses existing AmountRange type from src/types/trade.ts (unchanged)
function mapAmountRange(raw: string): AmountRange {
  // Strip non-numeric except commas, parse upper bound
  const numbers = raw.replace(/[^0-9,]/g, ' ').trim().split(/\s+/)
    .map(n => parseInt(n.replace(',', ''), 10)).filter(n => !isNaN(n))
  const upper = Math.max(...numbers)
  if (upper <= 15000) return '$1k-$15k'
  if (upper <= 50000) return '$15k-$50k'
  if (upper <= 100000) return '$50k-$100k'
  if (upper <= 250000) return '$100k-$250k'
  if (upper <= 500000) return '$250k-$500k'
  if (upper <= 1000000) return '$500k-$1M'
  return '$1M+'
}
```

## Open Questions

1. **Exact Alva SDK module name for senator trades**
   - What we know: Alva's `equity_ownership_and_flow` partition covers "senator trading activity"; the project context references `getSenatorTrades` as the function name
   - What's unclear: The precise `require()` path inside the V8 sandbox (e.g., `@arrays/equity/senator-trades:v1.0.0` is a guess)
   - Recommendation: As the first action in the rewrite script, call `GET /api/v1/sdk/partitions/equity_ownership_and_flow/summary` with the live API key to discover the exact module name and function signatures. If that endpoint returns insufficient detail, call `GET /api/v1/sdk/doc?name=<module>`.

2. **Alva senator trade response field names**
   - What we know: Congressional trade APIs consistently return name, ticker, transaction date, filing date, transaction type, and amount range. The exact field names in Alva's response are undocumented publicly.
   - What's unclear: Whether Alva returns `"transactionDate"` vs `"trade_date"`, `"Purchase"` vs `"buy"`, party affiliation, state, and chamber directly vs requiring enrichment from Congress.gov
   - Recommendation: Plan for a defensive mapping layer that handles multiple possible field name formats. Log the first raw Alva response to a debug file (`public/data/_alva-raw-sample.json`) during initial development run so the actual shape is visible.

3. **Pagination handling for full congressional dataset**
   - What we know: Alva's `/api/v1/run` endpoint executes a script that returns results; large datasets may need pagination
   - What's unclear: Whether the senator trades SDK returns all trades in one call or requires offset/limit pagination
   - Recommendation: D-01 says pull ALL trades. Implement a `hasMore` check in the mapping loop. If the response includes pagination metadata, loop until exhausted. Set a maximum of 5000 trades as a sanity cap with a warning.

4. **Whether Alva requires both House AND Senate separately**
   - What we know: `getSenatorTrades` implies senators only; House members file STOCK Act disclosures separately
   - What's unclear: Does Alva have a separate `getRepresentativeTrades` function? Does the senator trades function cover both chambers?
   - Recommendation: If only senator trades are available, the politician roster will be Senate-only. That is valid per D-02 (any politician returned by Alva with trades). Document clearly in build output: "Sourced from Alva — covers [Senate/Congress]."

5. **S&P 500 benchmark period for return calculation**
   - What we know: D-05 uses Alva for both trades and stock prices; existing `RawTrade` shape has `sp500Return` field; current simulated approach generated SP500 returns independently
   - What's unclear: Should SP500 return be computed per-trade (from trade date to today), or for a fixed season window?
   - Recommendation: Per-trade calculation is more accurate. For each trade, fetch SPY price from `tradeDate` to `min(tradeDate + 365days, today)` via Alva OHLCV. This matches how return-vs-benchmark is computed in real financial analysis.
