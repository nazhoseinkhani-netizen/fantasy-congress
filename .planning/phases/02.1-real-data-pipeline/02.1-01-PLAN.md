  <action>
  Complete rewrite of `scripts/fetch-trades.ts`. Remove ALL simulated trade generation code (mulberry32 RNG, normalRandom, weightedChoice, generateTradesForPolitician, getPreferredSectors, tryQuiverAPI). Replace with Alva REST API integration.

  The rewritten script must do the following in order:

  **1. Alva API setup and validation:**
  - Read `ALVA_API_KEY` from `process.env.ALVA_API_KEY`. If not set, print `[fetch-trades] FATAL: ALVA_API_KEY not set — build cannot proceed. Set ALVA_API_KEY in .env or environment.` and `process.exit(1)`.
  - Read `ALVA_ENDPOINT` from `process.env.ALVA_ENDPOINT`, default to `'https://api-llm.prd.alva.ai'`.
  - Create helper function `callAlvaScript(code: string): Promise<unknown>` that POSTs to `${ALVA_ENDPOINT}/api/v1/run` with headers `{ 'Content-Type': 'application/json', 'X-Alva-Api-Key': ALVA_API_KEY }` and body `{ code }`. On non-OK response, throw Error with status and first 300 chars of response body.
  - Add simple retry logic: if a call fails with 429 or 5xx, retry up to 2 more times with 2-second delay between attempts.

  **2. SDK discovery (first run diagnostic):**
  - Before fetching trades, call `GET ${ALVA_ENDPOINT}/api/v1/sdk/partitions/equity_ownership_and_flow/summary` with the API key header to discover the exact module name for senator trades.
  - Log the discovery response to console: `[fetch-trades] SDK partition discovery: ${JSON.stringify(result).slice(0, 500)}`
  - Also write the raw discovery response to `public/data/_alva-sdk-discovery.json` for debugging.
  - If discovery fails, log a warning but continue — try the inferred module name.

  **3. Fetch senator trades from Alva:**
  - Call `callAlvaScript()` with an inline script that uses `require("@arrays/equity/senator-trades:v1.0.0")` (or the discovered module name) to call `getSenatorTrades({})` with no time filter (per D-01: pull ALL available trades).
  - If the exact module path from discovery differs, use the discovered path instead.
  - Write raw Alva response to `public/data/_alva-raw-trades.json` for debugging.
  - If the response has pagination metadata (e.g. `hasMore`, `nextOffset`), loop until all trades are fetched. Set a sanity cap of 10000 trades with a warning.
  - If the API returns 0 trades or fails entirely, print `[fetch-trades] FATAL: Alva returned 0 senator trades — check API key and endpoint` and `process.exit(1)` (per D-04: build fails loudly, no fallback).

  **4. Politician name normalization and lookup:**
  - Load `_raw-politicians.json` from `public/data/_raw-politicians.json`.
  - Build a `Map<string, RawPolitician>` keyed by normalized name. Normalization function: strip honorifics (Rep., Sen., Dr., Mr., Mrs., Ms.), handle "Last, First" by splitting on comma and reversing, lowercase, trim. Also build a secondary lookup by last name only for fuzzy matching.
  - For each Alva trade, normalize the politician name and look up in the map to get `bioguideId`, `party`, `chamber`.
  - If a politician name from Alva doesn't match any Congress.gov politician, log a warning `[fetch-trades] WARNING: No match for "${alvaName}" — skipping` and skip that trade. Count and report total unmatched at the end.

  **5. Fetch stock prices from Alva for return calculations:**
  - Collect all unique tickers from the fetched trades.
  - Also add `SPY` to the ticker list (S&P 500 benchmark).
  - For each unique ticker, call `callAlvaScript()` with an inline script using `require("@arrays/data/stock/ohlcv:v1.0.0")` (or discovered module from `spot_market_price_and_volume` partition) to get OHLCV data. Request from `tradeDate` to `min(tradeDate + 365 days, today)` with interval `"1d"`.
  - Batch price lookups: process up to 5 tickers concurrently using Promise.allSettled with a semaphore pattern (simple queue with 5 concurrent max).
  - Cache results in a `Map<string, { date: string; close: number }[]>` so each ticker is fetched only once.
  - Per D-06: if price lookup fails for a ticker, log warning and mark affected trades. Those trades will have `absoluteReturn: 0`, `sp500Return: 0`, `returnVsSP500: 0` (kept in dataset but excluded from scoring by score-all.ts since returns are 0 which is valid, or NaN which gets excluded).

  **6. Compute returns:**
  - For each trade, compute `absoluteReturn`: `((latestPrice - tradePrice) / tradePrice) * 100` where `tradePrice` is the close price on or nearest to `tradeDate`, and `latestPrice` is the close price on or nearest to `min(tradeDate + 365d, today)`.
  - Compute `sp500Return` the same way using SPY prices for the same date range.
  - Compute `returnVsSP500 = absoluteReturn - sp500Return`.
  - If price data is unavailable for a trade's ticker, set all three returns to `0` and `daysToDisclose` to `0` if filing date is also missing.

  **7. Ticker enrichment (company name and sector):**
  - Keep the existing `TICKERS` static map (the `Record<string, TickerInfo>` with ~60 entries already in the file). This is the primary enrichment source.
  - For tickers not in the static map: set `company` to the ticker symbol (e.g., `"PLTR"` becomes company `"PLTR"`), set `sector` to `"Other"`. Log a warning listing all unknown tickers at the end.
  - Do NOT call Alva for sector enrichment — static map + "Other" fallback is simpler and faster (per research: don't hand-roll what the static map already covers).

  **8. Map to RawTrade and write output:**
  - For each Alva trade that matched a politician, construct a `RawTrade` object:
    - `id`: `${bioguideId}-${ticker}-${tradeDate}-${index}` (same pattern as before)
    - `bioguideId`, `politicianName`, `party`, `chamber`: from the politician lookup
    - `ticker`: from Alva trade data
    - `company`: from TICKERS map or ticker symbol fallback
    - `sector`: from TICKERS map or "Other" fallback
    - `tradeType`: normalize Alva's transaction type — if lowercase includes "purchase" or "buy" then "buy", else "sell"
    - `disclosureDate`: from Alva's filing/disclosure date field (ISO date string)
    - `tradeDate`: from Alva's transaction date field (ISO date string)
    - `amountRange`: parse Alva's amount string using `mapAmountRange()` function. Implementation: strip non-numeric except commas, parse all numbers, take the max (upper bound), map to AmountRange union: `<= 15000 -> '$1k-$15k'`, `<= 50000 -> '$15k-$50k'`, `<= 100000 -> '$50k-$100k'`, `<= 250000 -> '$100k-$250k'`, `<= 500000 -> '$250k-$500k'`, `<= 1000000 -> '$500k-$1M'`, `> 1000000 -> '$1M+'`. If parsing fails, default to `'$1k-$15k'`.
    - `returnVsSP500`, `absoluteReturn`, `sp500Return`: computed in step 6
    - `daysToDisclose`: `Math.floor((new Date(disclosureDate).getTime() - new Date(tradeDate).getTime()) / (1000 * 60 * 60 * 24))`. If negative or NaN, set to 0.
  - Write the array to `public/data/_raw-trades.json` using `writeFileSync` with `JSON.stringify(trades, null, 2)`.

  **9. Build summary:**
  - Log summary: total Alva trades received, matched to politicians, unmatched (skipped), price lookups succeeded/failed, unique tickers, unique politicians with trades, trades with missing returns.
  - If matched trades < 10, print `[fetch-trades] FATAL: Only ${count} trades matched politicians — insufficient data` and `process.exit(1)`.

  **Important constraints:**
  - Use relative imports for any project code (e.g., `../src/lib/...`) not `@/` aliases — tsx doesn't resolve tsconfig paths.
  - Keep the existing `TICKERS` map in the file (it's useful for enrichment). Remove everything else from the old file (RNG, simulation functions, Quiver code).
  - The Alva SDK `require()` calls run inside Alva's V8 sandbox, not in Node.js — they go inside the `code` string passed to `callAlvaScript()`.
  - Handle defensive field mapping for Alva responses — the exact field names (transactionDate vs trade_date, Purchase vs buy) are uncertain. Log the first raw record for debugging. Try multiple common field name patterns.
  </action>
