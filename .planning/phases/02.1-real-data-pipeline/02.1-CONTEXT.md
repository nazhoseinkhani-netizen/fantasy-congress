# Phase 2.1: Real Data Pipeline - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace all simulated trade data with real Congressional stock disclosures from the Alva Skills API (`getSenatorTrades`), fetch real stock prices for return calculations using Alva stock data, recompute all scoring from real data, and regenerate demo league matchups from real scores. The app shows actual Congressional corruption, not fake data. Demo league structures (rosters, matchups, weekly results) remain the only simulated layer.

</domain>

<decisions>
## Implementation Decisions

### Data Scope & Season
- **D-01:** Pull ALL available trades from Alva `getSenatorTrades` — no time filtering. More data = richer profiles, better scoring distributions, more realistic feel.
- **D-02:** Politician roster driven entirely by real data — any politician returned by Alva with trades becomes a player. No minimum count requirement, no padding with non-traders.
- **D-03:** Fantasy "season weeks" mapped to the most recent 3 months of trade data, divided into 6 weekly matchup windows. Matchup scores based on trades that landed in each week window.

### Fallback Strategy
- **D-04:** Build fails loudly if Alva API is unreachable — no simulated data ever ships. Clear error message, forces API access to be fixed before deploying. Matches Phase 1 D-01 intent. Remove the existing simulated trade generation fallback from `fetch-trades.ts`.

### Return Calculations
- **D-05:** Use Alva SDK for both trades AND stock prices — single data source for consistency. PROJECT.md lists stock prices as an Alva data source.
- **D-06:** Trades with missing or unavailable price data are excluded from scoring but kept in the dataset. Trade appears in the feed with 0 fantasy points. Politician still shows with scores from valid trades only. Matches Phase 1 D-07 approach.

### Data Mapping
- **D-07:** Enrich trades at build time — use ticker symbol to look up company name and sector from Alva stock fundamentals or a static mapping. Build script fills in all gaps before scoring runs.
- **D-08:** Keep existing TypeScript interfaces untouched. Build a mapping layer in the pipeline scripts that transforms Alva data into the existing `RawTrade`/`Politician` shape. All downstream code (scoring engine, UI components, data loaders) works without changes.

### Claude's Discretion
- Alva SDK initialization and authentication pattern
- Exact API call sequence and pagination handling
- Static ticker-to-sector/company mapping implementation
- S&P 500 benchmark data sourcing (Alva or calculated)
- Rate limiting and retry logic for Alva API calls
- Build script error reporting format
- How to determine "most recent 3 months" boundary from available data
- Whether to batch API calls or stream results

### Folded Todos
None — no pending todos matched this phase.

- **"Real Congressional corruption, not fake data"** — The user's core intent. Every trade in the app should be a real STOCK Act disclosure. The only fiction is the fantasy league matchup structure.
