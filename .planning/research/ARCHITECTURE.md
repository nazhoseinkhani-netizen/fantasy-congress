### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| Pages (app/) | Route entry points, Server Components that read from /data/ at build time | Next.js App Router pages with generateStaticParams |
| Data Layer (/data/) | Static JSON files committed to repo or generated by build script | JSON files from Alva API + data enrichment scripts |
| Scoring Engine (lib/scoring/) | Pure functions: trade data in, point totals out | Stateless TypeScript module, no side effects |
| Draft Simulator (lib/draft/) | Deterministic AI pick logic for simulated snake draft | Pure functions seeded from league config JSON |
| Zustand Stores (stores/) | User-facing game state: active team, draft picks, bench/active toggles | Zustand with persist middleware → localStorage |
| Share Card Generator (features/share/) | Render styled React component off-screen, export as PNG via html-to-image | React ref + html-to-image library |
| Data Access Module (lib/data/) | Typed helpers to read /public/data/*.json files at build or runtime | Typed TypeScript fetch wrappers |
| UI Components (components/) | Shared design system: cards, badges, tabs, tables | Tailwind CSS, no state of their own |
| Developer Mode Easter Egg (features/dev-mode/) | Overlay that reveals Alva API calls behind each data point | Hidden keyboard shortcut, Client Component |

### Structure Rationale

- **app/:** Route-first organization matches Next.js App Router conventions. Each route is a thin Server Component that reads from /public/data/ at build time.
- **lib/:** Pure logic isolated from React. Scoring engine and draft simulator here are easily unit-tested with no component dependencies.
- **features/:** Self-contained modules that combine components + logic for complex UI flows (draft room, share cards). Higher coupling tolerated here.
- **stores/:** Zustand stores are separate from both features and lib. They hold user game state that must survive page navigation (roster picks, draft session).
- **public/data/:** Static JSON is the "database." Structured for both full-list queries (index.json) and individual lookups ([bioguideId].json). Committed to repo or generated by scripts/ at build time.
- **scripts/:** Build-time data pipeline is separate from the app. These run once before `next build` to hydrate /public/data/. They're not bundled.

## Architectural Patterns

### Pattern 1: Static Data Layer — JSON Files as the Database

**What:** All application data lives in /public/data/ as typed JSON files. Pages read this data at build time via Server Components or via client-side fetch. No API routes needed at runtime.

**When to use:** Always, for this app. The "backend" is Alva's API, but it's called offline (via scripts/) not at request time.

**Trade-offs:** Build time increases proportionally with politician count (150 profile pages). The payoff is zero cold-start latency, no runtime API failures, and full CDN distribution. Stale data is an accepted trade-off for a prototype.

**Example:**
```typescript
// app/politicians/[bioguideId]/page.tsx
import { getPoliticianProfile } from '@/lib/data/politicians'

export async function generateStaticParams() {
  const politicians = await import('@/public/data/politicians/index.json')
  return politicians.default.map((p) => ({ bioguideId: p.bioguideId }))
}

export default async function PoliticianProfilePage({
  params,
}: {
  params: { bioguideId: string }
}) {
  const profile = await getPoliticianProfile(params.bioguideId)
  return <PoliticianProfile data={profile} />
}
```

### Pattern 2: Pure Scoring Engine — Functions Over Data

**What:** The scoring engine is a set of pure functions in lib/scoring/. It accepts raw trade data and scoring config, returns computed point totals. It has no I/O, no React, no side effects.

**When to use:** The scoring engine is called in two places: build scripts (to pre-compute season scores into results.json) and client-side (to show live breakdown in the UI). Pure functions work identically in both contexts.

**Trade-offs:** Pre-computing scores at build time means users see instant results. The trade-off is that "live" score updates require a rebuild. For a prototype, rebuilding on demand (or nightly) is acceptable.

**Example:**
```typescript
// lib/scoring/engine.ts
import type { Trade, ScoringConfig } from '@/types'

export function computeTradeScore(trade: Trade, config: ScoringConfig): number {
  let score = config.basePoints
  if (trade.returnPct > 0.1) score += config.bonuses.tenPctGain
  if (trade.disclosureLagDays > 45) score += config.penalties.lateDisclosure
  return score
}

export function computeRosterScore(
  trades: Trade[],
  rosterIds: string[],
  config: ScoringConfig
): number {
  return trades
    .filter((t) => rosterIds.includes(t.politicianId))
    .reduce((sum, t) => sum + computeTradeScore(t, config), 0)
}
```

### Pattern 3: Zustand with localStorage Persistence for Game State

**What:** User-facing game state (active roster, bench picks, league context) lives in Zustand stores with the persist middleware writing to localStorage. This gives the prototype the feel of real user accounts without any backend.

**When to use:** Any state that must survive page navigation or browser refresh: team roster selections, completed draft picks, preferred league view.

**Trade-offs:** localStorage is limited to ~5MB and is synchronous. This is fine for the prototype's data volume. The risk is hydration mismatch on first render (server sees no localStorage; client sees persisted state) — mitigated by using Zustand's built-in hydration skip on SSR.

**Example:**
```typescript
// stores/team.store.ts
import { create } from 'zustand'
import { persist } from 'zustand/middleware'

interface TeamStore {
  activeRosterIds: string[]
  benchIds: string[]
  setActive: (id: string) => void
  setBench: (id: string) => void
}

export const useTeamStore = create<TeamStore>()(
  persist(
    (set) => ({
      activeRosterIds: ['P000001', 'P000002', 'P000003'], // pre-populated defaults
      benchIds: ['P000004', 'P000005'],
      setActive: (id) =>
        set((s) => ({ activeRosterIds: [...s.activeRosterIds, id] })),
      setBench: (id) =>
        set((s) => ({ benchIds: [...s.benchIds, id] })),
    }),
    { name: 'fantasy-congress-team' }
  )
)
```

### Pattern 4: Server Components as Data Readers, Client Components as Interactors

**What:** Page-level components are Server Components that read /public/data/ JSON during build. Interactive sub-components (filters, tabs, roster toggles) are Client Components that receive data as props or read from Zustand.

**When to use:** Default for all pages. Only add 'use client' when a component needs event handlers, browser APIs, or store access.

**Trade-offs:** Cleaner separation of "what data" (Server Component concern) from "how to interact with it" (Client Component concern). Reduces JavaScript bundle size for data-heavy list views.

## Data Flow

### Build-Time Data Pipeline

```
Alva Skills API + bioguide.congress.gov + FEC
    ↓
scripts/fetch-politicians.ts
scripts/fetch-trades.ts
    ↓
scripts/compute-scores.ts (runs scoring engine over all trades)
scripts/generate-league.ts (builds demo league fixtures)
    ↓
public/data/politicians/index.json
public/data/politicians/profiles/[bioguideId].json  (150+ files)
public/data/trades/feed.json
public/data/leagues/demo-league.json
public/data/scoring/results.json
    ↓
next build → generateStaticParams reads index.json → generates 150+ HTML pages
    ↓
out/ folder → deployed to Vercel/Netlify CDN
```

### Page Load Data Flow

```
User visits /politicians/P000001
    ↓
CDN serves pre-built P000001.html (no server hit)
    ↓
React hydrates → Client Components mount
    ↓
Zustand rehydrates from localStorage (team state, league context)
    ↓
Profile tabs render from props (data embedded in HTML at build time)
    ↓
User interacts → Client Component handlers → Zustand store updates → localStorage persists
```

### Scoring Display Flow

```
Zustand: activeRosterIds (user's current lineup)
    +
public/data/trades/feed.json (fetched client-side or embedded at build time)
    ↓
lib/scoring/engine.ts: computeRosterScore(trades, rosterIds, config)
    ↓
Scoring breakdown component renders points per politician
    ↓
Compared against public/data/scoring/results.json (pre-computed opponent scores)
    ↓
Matchup scoreboard displays win/loss
```

### Draft Room Data Flow

```
User initiates draft
    ↓
Draft Store initializes: load politician pool from public/data/politicians/index.json
    ↓
lib/draft/snake.ts: generates pick order for 6 teams
    ↓
User picks → Draft Store updates pickedByUser[], availablePool[]
    ↓
lib/draft/simulator.ts: AI teams make picks (seeded, deterministic per league)
    ↓
Draft Store: full draft board state
    ↓
On draft complete: Team Store updated with user's drafted roster
    ↓
localStorage persists both stores
```

### Share Card Generation Flow

```
User clicks "Share" on politician card or team
    ↓
Share modal opens (Client Component)
    ↓
Hidden ShareCard component renders styled card via React ref
    ↓
html-to-image converts DOM node to PNG blob
    ↓
Browser downloads PNG or invokes Web Share API
```

### Key Data Flows Summary

1. **Build-time enrichment:** External APIs → scripts → /public/data/ JSON (happens once, offline, before deploy)
2. **Page render:** /public/data/ JSON → Server Component props → static HTML (at build time via Next.js)
3. **User interaction:** Zustand stores ↔ Client Components ↔ localStorage (at runtime, no network)
4. **Scoring computation:** Trade data + roster IDs → pure scoring functions → point totals (both at build time and in-browser)
5. **Draft simulation:** League config → draft simulator → deterministic AI picks → team rosters (at runtime, in-browser)

## Scaling Considerations

This app is a static prototype. "Scaling" concerns are around data size and build complexity, not server load.

| Scale | Architecture Adjustments |
|-------|--------------------------|
| 100 politicians | Current architecture handles comfortably. Single index.json, 100 profile JSONs, 100 static HTML pages. Build time ~30s. |
| 500+ politicians | index.json gets large (~500KB+). Split into paginated chunks or use static search index (Fuse.js over pre-built index). |
| Real-time data | Requires moving off static export. Add Vercel Edge Functions or move to SSR with ISR. Alva API called at request time, not build time. |
| Real multiplayer | Requires backend database (Postgres/Supabase), real auth, WebSocket for live draft. Out of scope for prototype — but the Zustand store shape maps cleanly to a real backend schema when needed. |

### Scaling Priorities

1. **First bottleneck (build time):** 150 politician profile pages is manageable. If politician count grows to 500+, switch to client-side data loading from index.json rather than generating a static HTML per profile.
2. **Second bottleneck (data freshness):** Pre-cached JSON goes stale. For a real product launch, add ISR (requires server deployment) or a daily rebuild webhook from Alva's API.

## Anti-Patterns

### Anti-Pattern 1: Calling Alva API at Runtime from the Browser

**What people do:** Fetch from the Alva Skills API directly in React useEffect or SWR hooks, using an API key stored in environment variables.

**Why it's wrong:** Exposes the Alva API key in the browser (even NEXT_PUBLIC_ keys are client-visible). API rate limits affect individual users. App breaks if Alva API is down.

**Do this instead:** Call Alva API only in build-time scripts. Cache results to /public/data/. Deploy static JSON. API key never touches the browser.

### Anti-Pattern 2: Storing Politician Data in Zustand

**What people do:** Load all politician data into a Zustand store on app boot and use it as a global data store across all pages.

**Why it's wrong:** Bloats initial JS bundle. localStorage limit (5MB) will be hit. Re-fetches every boot. Defeats the purpose of static generation.

**Do this instead:** Keep politician data in /public/data/ JSON. Pages read it at build time via Server Components. Client Components that need filtering/searching do so against a lightweight index, not the full dataset.

### Anti-Pattern 3: Putting Game Logic in React Components

**What people do:** Write scoring calculations directly inside component render functions or useEffect hooks inline.

**Why it's wrong:** Logic becomes untestable, duplicated, and mixed with presentation. The scoring engine needs to run both in the browser (live breakdown) and in build scripts (pre-computed results) — coupling it to React makes the second use case impossible.

**Do this instead:** All scoring, corruption index, and draft simulation logic lives in lib/ as pure TypeScript functions. React components call these functions but do not contain them.

### Anti-Pattern 4: Skipping generateStaticParams for Profile Pages

**What people do:** Use dynamic routes without generateStaticParams and deploy, expecting Next.js to handle it.

**Why it's wrong:** With output: 'export', dynamic routes without generateStaticParams cause a build error. There is no fallback — every politician profile must be enumerated at build time.

**Do this instead:** generateStaticParams reads the politicians index.json at build time and returns all bioguideIds. Every profile page gets its own pre-built HTML file.

### Anti-Pattern 5: One Monolithic JSON File for All Data

**What people do:** Build a single `all-data.json` with everything: politicians, trades, scores, leagues.

**Why it's wrong:** A file covering 150 politicians with 6 weeks of trade history each will exceed 5MB. Next.js embeds JSON data into the HTML page it's used on — a giant file bloats every page.

**Do this instead:** Separate JSON files by entity type and by individual (index.json for lists, [id].json for detail). Pages only fetch what they render.

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| scripts/ → public/data/ | File system write (JSON) | Decoupled: scripts run independently of app |
| Server Components → public/data/ | import() or fs.readFileSync at build time | No runtime network calls |
| Client Components → Zustand stores | React hooks (useTeamStore, etc.) | Never pass store slices as props through many layers |
